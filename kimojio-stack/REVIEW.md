# REVIEW-SYNTHESIS.md

## Review Summary

`kimojio-stack` has credible single-thread stackful/io_uring building blocks, but the specialist evidence does not support treating it as production-ready or performance-proven against Tokio for low-latency, high-throughput thread-per-core application servers yet (REVIEW-performance.md:5; REVIEW-release-manager.md:6-9; REVIEW-performance.md:230-249). The highest-priority blockers are fd lifetime safety for unregistered async I/O, cancellable waiter lifecycle, externally staged wake handling at scope exit, cancellation/resource-retention behavior, task-churn metadata growth, and missing server-grade benchmark/release gates (REVIEW-security.md:5-25; REVIEW-correctness.md:7-51; REVIEW-edge-cases.md:35-58; REVIEW-performance.md:53-72; REVIEW-performance.md:7-26; REVIEW-release-manager.md:75-95).

## Overall Assessment

- **Reliability:** Not release-ready because direct high-confidence findings show safe async I/O can lose fd identity, timed waits and cross-thread waits can leave stale waiter state, and scope exit can cancel a task that was externally woken but not yet drained into the scheduler ready queue (REVIEW-security.md:11-25; REVIEW-correctness.md:13-24; REVIEW-correctness.md:37-51; REVIEW-edge-cases.md:41-58).
- **Performance:** Promising substrate, not proven server runtime, because the benchmark suite is mostly microbenchmarks and specialists identified task-churn retention, per-completion `mincore`, waiter-queue O(N) wakeups, ready-batch CQE delay, cancellation leaks, and missing p99/p99.9 application-server comparisons (REVIEW-performance.md:5; REVIEW-performance.md:7-49; REVIEW-performance.md:76-116; REVIEW-performance.md:230-249; REVIEW-testing.md:107-125).
- **Fit for thread-per-core application servers:** Fit is currently as a low-level local-runtime substrate with explicit sharding and channel-only interop, not as a drop-in Tokio replacement with a first-class multi-worker owner, ambient async interop, or proven end-to-end latency behavior (REVIEW-assumptions.md:111-159; REVIEW-architecture.md:113-118; REVIEW-release-manager.md:97-116).
- **Release posture:** The crate needs an explicit Linux/kernel/MSRV/readme/stability/performance contract before public production positioning, because release specialists found CI and benchmarks exist but platform, MSRV, crate README, benchmark baselines, and alpha policy are not yet encoded as release artifacts (REVIEW-release-manager.md:6-9; REVIEW-release-manager.md:11-31; REVIEW-release-manager.md:33-73; REVIEW-release-manager.md:75-116).

## Must-Fix Findings

1. **Unregistered async I/O can outlive the borrowed fd identity.** Grounding: **Direct**. Final severity: **Must-fix**. Confidence: **High**. `read_async`/`write_async` copy a raw fd from a borrowed `AsFd`, returned `IoResult` does not carry an fd owner or lifetime, and submission may still be only queued in userspace, so safe caller code can close/reuse the numeric fd before the kernel consumes the SQE (REVIEW-security.md:11-25). This is the strongest safety blocker because the specialist warrant ties it to cross-connection data disclosure or corruption in application servers without unsafe caller code (REVIEW-security.md:18-20).

2. **Wait registration lifecycle is not cancellation-safe.** Grounding: **Direct**. Final severity: **Must-fix**. Confidence: **High**. Timed `wait_any`/`wait_all` register the task with waitables and return on timeout without unregistering, while `wake_one` consumers can spend a finite wake on stale task state and strand a live waiter (REVIEW-correctness.md:13-27; REVIEW-edge-cases.md:87-108; REVIEW-security.md:33-46). Stackful cross-thread `send`/`recv` move an `ExternalWaiter` into the channel wait queue before parking, but scope cancellation can unwind the task without removing that waiter, leaving external-wake liveness and notification FIFO state corrupted (REVIEW-correctness.md:37-51; REVIEW-edge-cases.md:13-31; REVIEW-assumptions.md:13-29).

3. **Scope quiescence ignores externally staged ready tasks.** Grounding: **Direct**. Final severity: **Must-fix**. Confidence: **High**. `wait_for_scope` decides to cancel remaining tasks from scheduler `ready_len()`, while external wakes first sit in `ExternalWakeInner.ready` and are drained only by `drive_scheduler`; therefore a just-woken cross-thread receiver can be classified as blocked and canceled before consuming its message (REVIEW-edge-cases.md:41-58).

4. **Pending async I/O drop/cancel is not a server-safe normal cancellation path.** Grounding: **Direct + Inferential**. Final severity: **Must-fix for production-server positioning**. Confidence: **High**. The docs and implementation intentionally leak backing buffers for pending dropped/canceled `IoResult`s, submit best-effort cancellation, and cannot reclaim the buffer after the handle is marked taken, making timeout/disconnect storms a memory-growth and extra-SQE/CQE traffic risk (REVIEW-performance.md:53-72; REVIEW-assumptions.md:33-55; REVIEW-testing.md:67-85).

5. **Long-lived scopes retain completed task metadata and runtime task slots.** Grounding: **Inferential from direct code evidence**. Final severity: **Must-fix for high-churn servers**. Confidence: **High**. Scope state keeps task ids and panic payload holders after completion, scheduler task slots grow monotonically, and the performance specialist ties this to O(total spawned) memory in long-lived per-connection/request scopes rather than O(live tasks) memory (REVIEW-performance.md:13-26).

6. **Fixed-resource lifetime safety needs release-blocking pending-drop coverage.** Grounding: **Contextual test-gap evidence**. Final severity: **Must-fix before release qualification**. Confidence: **High**. Tests cover happy-path registered async I/O and immediate slot reuse, but not the high-risk path where registered fd/buffer handles or pending `IoResult`s are dropped before the CQE, so regressions in lease attachment or delayed slot reuse would not be caught (REVIEW-testing.md:7-25).

7. **Platform and performance claims are not release-contract-ready.** Grounding: **Contextual**. Final severity: **Must-fix before publishing or production adoption**. Confidence: **High/Medium by subtopic**. The platform/kernel contract is not encoded despite unconditional io_uring dependencies and runtime panics on `IoUring::build`, while benchmark baselines are not release-gated and the current suite does not establish tail-latency or throughput superiority over Tokio for application-server workloads (REVIEW-release-manager.md:11-31; REVIEW-release-manager.md:75-95; REVIEW-performance.md:230-249; REVIEW-testing.md:107-125).

## Should-Fix Findings

1. **Barrier cancellation corrupts future generations.** Grounding: **Direct**. Final severity: **Should-fix**. Confidence: **High**. `Barrier::wait` increments `arrived` before parking and has no cancellation guard, so a canceled parked participant remains counted and can release later generations early or deadlock them (REVIEW-correctness.md:55-73; REVIEW-edge-cases.md:62-83).

2. **`Timeout::update` can target a recycled `IoState`.** Grounding: **Direct**. Final severity: **Should-fix**. Confidence: **High**. `try_wait` removes and recycles timeout state without invalidating `user_data`, while `update(&self)` can submit an update against that stale pointer after another timeout reuses the pooled `IoState` (REVIEW-correctness.md:77-95).

3. **Full-SQ recovery can panic on transient kernel submit errors.** Grounding: **Direct**. Final severity: **Should-fix**. Confidence: **High**. `submit_io` retries a failed SQE push after `submit_and_wait(0)`, but that submit path maps `EAGAIN`/`EBUSY`/`EINTR` to zero submitted entries, so the second push can still fail and panic (REVIEW-edge-cases.md:112-133).

4. **Fixed-resource unregister failures are ignored before slot reuse.** Grounding: **Direct**. Final severity: **Should-fix**. Confidence: **Medium**. Fixed fd and buffer unregister paths ignore update errors while returning slots to free lists, which can diverge runtime slot state from kernel fixed-resource tables (REVIEW-edge-cases.md:137-159).

5. **Scheduler and waiter hot paths need latency-oriented cleanup.** Grounding: **Inferential with direct hot-path evidence**. Final severity: **Should-fix**. Confidence: **High**. Specialists identified unconditional `mincore` on every coroutine completion, O(waiters) `Vec::remove(0)` wake-one queues, default ready-batch CQE delay, and missing scheduler fairness tests around `AfterReadyTasks` (REVIEW-performance.md:30-49; REVIEW-performance.md:76-116; REVIEW-testing.md:27-45).

6. **Cross-thread channel hot paths need clearer fast-path and blocking policies.** Grounding: **Inferential with direct hot-path evidence**. Final severity: **Should-fix**. Confidence: **High/Medium**. Ready send/recv paths lock wait-queue mutexes even when waiter counts are zero, and stackful blocking performs fixed spin/yield retries before parking, both of which can add CPU cost and jitter under high message or miss rates (REVIEW-performance.md:120-160).

7. **Root-vs-coroutine capability boundaries should be represented explicitly.** Grounding: **Direct API-shape evidence**. Final severity: **Should-fix**. Confidence: **High/Medium**. The same `RuntimeContext` type is accepted by root-only, coroutine-only, and shared APIs, and cross-thread stackful endpoint misuse from root can park and panic as a deadlock instead of failing at the type or targeted-error boundary (REVIEW-architecture.md:37-59; REVIEW-security.md:48-67).

8. **Core runtime layering and file organization should be split before more features land.** Grounding: **Contextual architecture/maintainability evidence**. Final severity: **Should-fix**. Confidence: **High**. Scheduler policy, io_uring driving, registered resources, external wake plumbing, public APIs, unsafe contracts, allocation tracking, and tests are concentrated in `src/lib.rs`, while future multiring/worker-local design notes expect clearer scheduler/I/O/wake/resource seams (REVIEW-architecture.md:7-33; REVIEW-maintainability.md:27-43).

9. **Documentation and packaging should match the stackful crate.** Grounding: **Contextual**. Final severity: **Should-fix**. Confidence: **High/Medium**. `kimojio-stack` inherits the workspace async-runtime README, several public synchronization primitives lack module-level examples/semantic notes, and release documentation does not yet present a crate-specific quickstart, limitations, stability, or platform contract (REVIEW-maintainability.md:7-23; REVIEW-maintainability.md:47-63; REVIEW-release-manager.md:54-73).

10. **CI/test gates should cover cancellation, closure, waiter, and benchmark rot cases.** Grounding: **Contextual test-gap evidence**. Final severity: **Should-fix**. Confidence: **High/Medium**. Specialists found gaps in scheduler fairness, local primitive close/drop readiness, dropped async I/O drain outcomes, allocation budgets for contended paths, benchmark smoke gates, and MSRV metadata alignment (REVIEW-testing.md:27-105; REVIEW-release-manager.md:33-52; REVIEW-release-manager.md:75-95).

## Consider

- **Clarify scope semantics as cancel-on-quiescence.** Grounding: **Direct + Contextual**. The docs compare scope to `std::thread::scope` while implementation cancels blocked unjoined children when no runnable work remains, so examples and naming should make that structured-concurrency policy obvious (REVIEW-architecture.md:63-85).
- **Track cross-thread `Waitable` integration as a design gap, not a quick patch.** Grounding: **Contextual**. Local channels, timers, joins, and I/O handles share `Waitable`, but cross-thread stackful endpoints intentionally do not because cancellation-safe external waiter removal is not yet available (REVIEW-architecture.md:89-111).
- **Avoid successful waits blocking on timeout cleanup when timeout-remove is unsupported.** Grounding: **Direct + Conditional**. `cancel_timeout` can wait for the original timeout when `IORING_OP_TIMEOUT_REMOVE` is unavailable, so an already-successful `wait_any`/`wait_all` may still pay the timeout duration on that support matrix (REVIEW-correctness.md:99-115).
- **Capacity-model stackful memory and stack sizing.** Grounding: **Inferential**. Default 64 KiB guarded stacks, fixed stack sizing, and per-coroutine stack allocation require VMS/RSS/VMA/page-table capacity modeling and documentation for high-concurrency servers (REVIEW-performance.md:208-226; REVIEW-assumptions.md:85-107; REVIEW-edge-cases.md:163-185).
- **Document the multi-runtime sharding recipe.** Grounding: **Contextual**. The crate exposes a single local `Runtime`, while the thread-per-core model assumes applications create, pin, shard, and coordinate per-core runtimes and keep I/O ownership local (REVIEW-assumptions.md:111-133).
- **Keep Tokio interop explicitly channel-only unless the design changes.** Grounding: **Contextual**. The crate says it is not an async runtime, and Tokio-compatible integration is limited to `channel::cross_thread` futures rather than ambient driving of arbitrary futures or runtime I/O (REVIEW-assumptions.md:137-159).
- **Measure registration churn and `wait_any` fan-in limits.** Grounding: **Inferential**. Fixed resource registration only pays off when amortized, and `wait_any`/`wait_all` linearly scan/register all waitables while lacking removal, so server docs should bound these APIs or benchmark their intended scale (REVIEW-performance.md:164-204).
- **Keep unsafe-aliasing documentation adjacent to lock guard dereferences.** Grounding: **Contextual maintainability**. `Mutex`/`RwLock` use `UnsafeCell` with runtime counters but do not document the local safety invariants at dereference sites (REVIEW-maintainability.md:67-83).

## Trade-offs Requiring Decision

- **Safety leak vs deterministic reclamation:** Leak-on-drop preserves memory safety for pending kernel-owned buffers, but the same design can convert ordinary timeout/disconnect cancellation into unbounded memory growth unless a reaping API or strict buffer-pool discipline becomes the normal path (REVIEW-performance.md:53-72; REVIEW-assumptions.md:39-55).
- **Throughput batching vs CQE tail latency:** `AfterReadyBatch` batches ready tasks before nonblocking ring entry, which can improve throughput, but specialists flagged linear CQE-to-resume delay behind large ready batches and recommended latency budgets/metrics for server workloads (REVIEW-performance.md:98-116).
- **Stackful ergonomics vs capacity:** Fixed guarded stacks simplify synchronous-looking code and trap overflow, but high concurrency requires explicit stack-size tuning and capacity planning against async task footprints (REVIEW-performance.md:208-226; REVIEW-assumptions.md:91-107).
- **Single-file locality vs evolvable seams:** Keeping the core in one module may preserve review locality, but specialists found scheduler, io_uring, resource, external-wake, and API responsibilities already interleaved against future worker-local/multiring design goals (REVIEW-architecture.md:21-33; REVIEW-maintainability.md:33-43).
- **Fail-fast panics vs operator-facing errors:** Runtime creation, root-context stackful misuse, stack-size overflow/allocation failure, and some SQE paths currently panic or can panic, while server deployment and release specialists favored explicit error contracts or targeted misuse failures (REVIEW-maintainability.md:87-103; REVIEW-security.md:48-67; REVIEW-edge-cases.md:112-185).
- **Cross-thread waiter ergonomics vs cancellation safety:** Implementing cross-thread `Waitable` would improve heterogeneous waits, but specialists agree the current omission is safer until cancellation-safe external waiter removal exists (REVIEW-architecture.md:89-111; REVIEW-correctness.md:31-51).
- **Fixed resources as optimization vs connection churn:** Fixed fd/buffer APIs can improve steady-state I/O only when registration is amortized, while per-connection register/unregister churn needs measurement before server guidance can claim wins (REVIEW-performance.md:164-182).
- **Alpha experimentation vs adopter expectations:** Version `0.0.1-alpha.0` and future multithreading notes support experimentation, but release specialists warned public runtime APIs still need explicit no-stability or stable-core policy to avoid accidental production lock-in (REVIEW-release-manager.md:97-116).

## Observations

- The local-runtime architecture is coherent for a single-thread cost model: specialists cite pervasive local state and an explicit cross-thread channel boundary as a good fit for keeping hot paths local and cross-runtime behavior visible (REVIEW-architecture.md:113-118; REVIEW-assumptions.md:59-81).
- The crate already has useful release-readiness assets: CI runs clippy/tests/doctests/fmt/all-features clippy, rustdoc explains scheduler/io_uring/cancellation/allocation limitations, and Criterion benchmarks cover runtime/scheduler/sync/channel/io_uring micro-paths (REVIEW-release-manager.md:6-9).
- The allocation and fixed-resource design shows serious low-level intent, because warmed pipe loops and fixed-resource APIs are documented/benchmarked, even though cancellation, contention, churn, and tail-latency coverage remain insufficient for server claims (REVIEW-performance.md:5; REVIEW-testing.md:87-105; REVIEW-performance.md:230-249).
- Explicit channel-only async/Tokio interop is architecturally clear, but it should remain documented because downstream users may infer broader compatibility from Tokio-facing channel constructors (REVIEW-assumptions.md:137-159; REVIEW-architecture.md:115-117).

## Dissent Log

- The unregistered fd-lifetime finding would be rebutted only if the runtime can prove SQEs are submitted and the kernel has taken a file reference before the borrow ends, or if the API prevents fd drop/reuse while pending; the reviewed evidence did not show either condition (REVIEW-security.md:21-25).
- Stale timed waiters would be less severe if all `wake_one` paths validate live waiters or cascade past stale waiters, but specialists found no cancellation token, no targeted removal, and no validation in `Waiters::wake_one` (REVIEW-correctness.md:23-27; REVIEW-edge-cases.md:104-108).
- Stale cross-thread stackful waiters would be acceptable only if such waits cannot be canceled while parked or if channel drop/notification always follows cancellation, but specialists cite public scoped cancellation and persistent endpoints as counterevidence (REVIEW-correctness.md:47-51; REVIEW-assumptions.md:23-29).
- Leak-on-drop is a deliberate memory-safety trade-off and is less concerning if pending `IoResult`s are never dropped at meaningful volume or upper layers always reap completions before releasing buffers (REVIEW-performance.md:68-72; REVIEW-assumptions.md:49-55).
- Task metadata retention is less concerning for bounded long-lived coroutines or scopes that return promptly after children complete, but not for process-lifetime scopes spawning per connection/request (REVIEW-performance.md:19-26).
- Ready-batch scheduling may be the right throughput default when ready queues are small or users configure `AfterReadyTasks(n)`, but specialists require measurement and guidance before claiming low tail latency (REVIEW-performance.md:109-116).
- Module splitting is less urgent if the crate intentionally remains a small single-thread runtime or if a single-file core is a measured review-locality choice, but future worker-local/multiring notes and current file size argue for seams before growth (REVIEW-architecture.md:27-33; REVIEW-maintainability.md:39-43).
- Cross-thread `Waitable` omission is not criticized as unsafe; specialists preserve the dissent that implementing it before cancellation-safe external waiter removal would be worse than the current ergonomic gap (REVIEW-architecture.md:105-111).
- Platform and release-contract gaps are not blockers for unpublished/internal Linux-only use with external platform standards, but they are blockers for public production positioning (REVIEW-release-manager.md:27-31; REVIEW-release-manager.md:69-73).

## Synthesis Trace

- **Grounding key:** **Direct** means the specialist tied the finding to concrete code paths and a specific failure mode; **Inferential** means direct code evidence plus workload/server-performance reasoning; **Contextual** means packaging, docs, CI, benchmark, or maintainability evidence rather than a demonstrated runtime bug (REVIEW-security.md:11-25; REVIEW-performance.md:19-26; REVIEW-release-manager.md:24-31).
- **Wait lifecycle cluster:** Merged timed `wait_any`/`wait_all` stale registrations, cross-thread stackful waiter cancellation, and external-wake readiness staging because all affect whether a parked task registration is live, removable, and fairly woken (REVIEW-correctness.md:7-51; REVIEW-edge-cases.md:7-58; REVIEW-security.md:27-46; REVIEW-assumptions.md:7-29).
- **I/O lifetime/resource cluster:** Merged unregistered fd lifetime, pending async drop/cancel leaks, registered-resource pending-drop test gaps, and fixed-resource unregister failure handling because all concern kernel-visible resource identity and reclamation across SQE/CQE boundaries (REVIEW-security.md:5-25; REVIEW-performance.md:53-72; REVIEW-testing.md:7-25; REVIEW-edge-cases.md:137-159).
- **Scheduler/performance cluster:** Merged long-lived scope metadata retention, unconditional final stack usage measurement, ready-batch CQE delay, O(N) wake queues, and scheduler fairness test gaps because all influence per-core throughput, latency, or memory under high task/waiter churn (REVIEW-performance.md:7-49; REVIEW-performance.md:76-116; REVIEW-testing.md:27-45).
- **API/release/docs cluster:** Merged root-vs-coroutine capability shape, runtime layering, crate README mismatch, MSRV/platform/stability contracts, and primitive docs because they determine whether the crate is easy and safe to program as the stackful alternative it aims to be (REVIEW-architecture.md:37-59; REVIEW-architecture.md:7-33; REVIEW-maintainability.md:7-63; REVIEW-release-manager.md:11-116).
- **Performance-proof cluster:** Merged benchmark-suite limitations, lack of tail-latency contention coverage, and release perf-gate absence because the user’s evaluation question depends on measured low-latency high-throughput behavior versus Tokio, not only microbenchmarks (REVIEW-performance.md:230-249; REVIEW-testing.md:107-125; REVIEW-release-manager.md:75-95).
- **Confidence weighting:** Direct high-confidence code-path evidence was promoted even when only one specialist raised it, as with unregistered fd lifetime and external-ready scope cancellation, while multiple lower-severity contextual docs/test gaps were kept as should-fix or consider unless they gate release claims (REVIEW-security.md:5-25; REVIEW-edge-cases.md:35-58; REVIEW-maintainability.md:47-63; REVIEW-testing.md:87-125).
